# Database & Migrations Runbook

NestJS 11 + **Prisma 7** + PostgreSQL 16. This guide covers how the app talks to
the database, how migrations are authored and applied, the `db:*` commands, the
per-environment databases, seeding, and soft-delete discipline.

See also: [`./local-development.md`](./local-development.md) (full env model and
day-to-day dev loop), [`./commands.md`](./commands.md), [`./deploy.md`](./deploy.md)
(the prod `migrate` service), [`./backup-and-restore.md`](./backup-and-restore.md),
and [`./troubleshooting.md`](./troubleshooting.md).

## Prisma 7 setup (the adapter pattern)

This app does **not** use Prisma's default query engine. It connects through the
`@prisma/adapter-pg` driver adapter over an **explicit `pg` Pool** that the app
constructs and owns.

- `prisma.config.ts` (Prisma 7's config file, replacing the old `schema.prisma`
  datasource `url`) wires `schema`, `migrations.path`, and the datasource `url`
  from `DATABASE_URL`. It loads `dotenv/config` so CLI commands (`prisma migrate`,
  `prisma studio`) see the env.
- `src/common/prisma/prisma.service.ts` builds a `pg.Pool`
  (`max` = `DB_POOL_MAX` ?? 15, `connectionTimeoutMillis` 5000, `idleTimeoutMillis`
  30000, `statement_timeout` = `DB_STATEMENT_TIMEOUT_MS` ?? 30000), wraps it in
  `PrismaPg`, and passes that adapter to `super({ adapter })`. The app owns the
  pool (so `/metrics` can report live pool stats via `getPoolStats()`); it must
  `pool.end()` on shutdown, and it logs idle-pool client errors rather than
  letting `pg` crash the process.

### `prisma.client` vs `$queryRaw`

`PrismaService` extends `PrismaClient` but also exposes a **`.client`** property —
the soft-delete-extended client (`applySoftDelete(this)`, a `$extends` result).

- **All model data access goes through `prisma.client.<model>`**
  (e.g. `prisma.client.account.findMany(...)`). This is what applies the
  `deleted_at` auto-filter and the hard-delete guards.
- **Raw SQL** uses `prisma.$queryRaw` / `prisma.$executeRaw` (on the base service,
  not `.client`). Raw SQL bypasses the soft-delete extension, so raw queries must
  add `deleted_at IS NULL` themselves where it matters (see the trigram search
  helper for the established pattern).
- `PrismaModule` is `@Global`, so `PrismaService` is injectable everywhere without
  re-importing.

## HAND-AUTHORED migrations (critical)

> ⚠️ **Migrations in this repo are hand-written SQL, not blindly autogenerated.**
> Several carry DDL that `prisma migrate dev` will never produce on its own.

Migrations live at `prisma/migrations/<timestamp>_<name>/migration.sql`. The
`schema.prisma` is the source of truth for **models**, but the migration SQL is
reviewed and hand-edited — especially for ledger/numbering/trigger constructs that
the Prisma schema cannot express:

- **`journal_lines` one-sided CHECK** (`20260611022405_add_journal`):
  `CHECK (debit >= 0 AND credit >= 0 AND (debit = 0 OR credit = 0) AND (debit > 0 OR credit > 0))`
  — DB-level enforcement of double-entry (every line is exactly one-sided and
  non-zero). Not representable in `schema.prisma`.
- **`audit_log` append-only trigger** (`20260617000001_audit_log_append_only`):
  a `BEFORE UPDATE OR DELETE` trigger that `RAISE EXCEPTION`s, so the audit table
  is immutable even to the app/migrate DB role. INSERT/SELECT are unaffected.
- **Partial-unique index for singleton account roles**
  (`20260618000000_account_role`):
  `CREATE UNIQUE INDEX ... ON accounts (role) WHERE role IS NOT NULL AND role <> 'CASH'`
  — at most one account per singleton role; CASH may be held by many. The same
  migration includes a behavior-preserving `UPDATE` backfill of the seeded chart.
- **Trigram search** (`20260616000000_add_trigram_search`): `CREATE EXTENSION
  pg_trgm` + GIN `gin_trgm_ops` indexes on search columns.
- **Gapless numbering** relies on `journal_sequences` / `document_sequences` tables
  (created in these migrations) plus `SELECT ... FOR UPDATE` in the service layer —
  the schema defines the tables; the gapless guarantee is application + migration.

Note: the `@@unique(..., name: "...")` parameter in `schema.prisma` does **not**
rename the DB index — Prisma still emits the default index name (e.g.
`accounts_code_key`). The `name` is the Prisma-side accessor only.

### Authoring workflow

1. Edit `prisma/schema.prisma` (add/change models, enums, fields).
2. Generate the migration scaffold, then **review and hand-adjust the SQL** —
   add CHECK constraints, triggers, partial indexes, backfills, `CREATE EXTENSION`,
   etc. that the schema cannot express. For pure DDL that maps cleanly, the
   autogenerated SQL is fine; for the ledger/trigger/index cases above, write it
   by hand.
3. Run `npm run db:generate` to regenerate the typed client from the schema.
4. Verify with the **e2e suite** — `npm run test:e2e` spins up a Postgres 16
   Testcontainer and runs `npx prisma migrate deploy` against it
   (`test/testcontainers.ts`), so **every e2e run replays the full migration
   history from scratch**. A broken or non-idempotent migration fails the suite.

> ⚠️ **Do not let `prisma migrate dev` clobber a hand-authored migration.** If you
> edit the schema after hand-writing a migration's SQL, `migrate dev` may try to
> generate a *new* migration or, worse, prompt to reset. Keep schema edits and the
> matching hand-written SQL in the **same** migration directory; never let the CLI
> overwrite the custom SQL (CHECK/trigger/partial-index blocks). When in doubt,
> craft the migration directory and SQL manually and let `migrate deploy` apply it.

## Commands

| Command | What it does |
| --- | --- |
| `npm run db:migrate` | `prisma migrate dev` against `.env.development` (via `dotenv-cli`). **Local dev only** — creates/applies a dev migration and regenerates the client. |
| `npm run db:reset`   | `prisma migrate reset` — **DROPs the DB**, re-applies all migrations, re-seeds. **DEV ONLY; destroys all data.** |
| `npm run db:studio`  | `prisma studio` — browser GUI for the dev DB. |
| `npm run db:generate`| `prisma generate` — regenerate the typed client. **Run after ANY schema change.** |
| `npm run create-admin -- <email> <password> "<name>"` | Bootstrap the first ADMIN user (see Seeding). |

All `db:*` and `create-admin` scripts load `.env.development` via `dotenv-cli`, so
they target the local dev database.

> **Stale "property does not exist" editor errors** after a schema change almost
> always mean the client is stale — run `npm run db:generate`. The source of truth
> is `npm run typecheck` (`tsc --noEmit`), not the editor's cached diagnostics.

### Production migrations (out-of-process, never in-process)

Production migrations are applied by a dedicated, one-shot **`migrate` compose
service** that runs `npx prisma migrate deploy` to completion **before** the `api`
starts. The `api` service `depends_on` `migrate` with
`condition: service_completed_successfully`, so the app never runs migrations
itself. See [`./deploy.md`](./deploy.md):

```bash
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --build
# builds the image, runs `migrate` (prisma migrate deploy), THEN starts api/caddy/backup
```

Migrations are **forward-only** in prod. Rolling back the app image does not undo a
migration — prefer a corrective forward migration, or restore from backup (see
[`./deploy.md`](./deploy.md) Rollback and [`./backup-and-restore.md`](./backup-and-restore.md)).

## Seeding

Seeding is **idempotent and runs on application boot** via `OnModuleInit` hooks —
no separate seed command for the core reference data:

- **Chart of accounts** — `AccountsService.seedIfEmpty()` inserts the SAK-aligned
  chart from `src/ledger/accounts/chart-of-accounts.seed.ts` if the table is empty.
- **Accounting periods** — `PeriodsService.generatePeriods()` ensures the current
  fiscal year's 12 monthly periods exist (`fiscalYearStartMonth` from company
  settings).
- **Company settings** (`CompanyService.seedIfEmpty`) and **tax codes**
  (`TaxCodesService.seedIfEmpty`) seed on boot the same way.
- **First ADMIN user** — there is no registration endpoint; bootstrap it explicitly:
  ```bash
  npm run create-admin -- admin@acme.co 's3cret!' "Budi Admin"
  ```
  `scripts/create-admin.ts` argon2-hashes the password and upserts the user via the
  adapter-pg client. In prod, run it once against the deployed DB.

> **System accounts are identified by `account.role`, not by hardcoded codes.** The
> seeded chart assigns the `AccountRole` enum (`CASH`, `AR_CONTROL`, `AP_CONTROL`,
> `RETAINED_EARNINGS`, `OPENING_BALANCE_EQUITY`, `TAX_EXPENSE`). New code must look
> up system accounts by `role`, never by code string. `CASH` may apply to multiple
> accounts (Kas, Bank); the other five are singletons enforced by the partial-unique
> index above.

## Soft-delete discipline

Models listed in `SOFT_DELETE_MODELS` (`src/common/prisma/soft-delete.extension.ts`)
— currently **User, Account, JournalEntry, TaxCode, BusinessPartner, SalesInvoice,
PurchaseBill, Payment** — carry a `deleted_at` (and `deleted_by`) column and are
governed by the soft-delete extension on `prisma.client`:

- `find* / count / aggregate / groupBy / update / updateMany` auto-inject
  `deletedAt: null` — tombstoned rows are invisible and unwritable (a write to a
  tombstone matches 0 rows → `P2025` → 404).
- `delete / deleteMany / upsert` on these models **throw** (a programmer-error
  guard → 500). Deletes are **tombstones**: call `prisma.client.<model>.softDelete(where, deletedBy)`,
  which stamps `deleted_at`. No HTTP route performs a hard delete.
- Because raw SQL bypasses the extension, raw queries against these tables must
  filter `deleted_at IS NULL` explicitly.
